## Strategy

### Tier 1: Pulse `assume_` in implementations (44 total, ~30 unique after dedup)
These are runtime assertions in Pulse `fn` code. Most are array bounds or
simple invariant properties. The Complexity versions duplicate the base versions.

**Approach**: Strengthen loop invariants to carry the needed facts. Fix base .fst
first, then mirror changes to .Complexity.fst.

**Key insight**: Many `assume_` are about:
- `u < n` when u comes from stack → add `stack_all_valid` to loop invariant
- `vtop < n` when discovering → follows from `vtop < n` before push
- `SZ.fits` overflow → need `n * n` fits precondition
- Final postconditions → need full DFS/BFS correctness invariants
